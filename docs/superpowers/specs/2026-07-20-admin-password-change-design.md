# 管理员修改密码设计

## 目标

为已登录管理员提供可发现、可验证的修改密码入口。管理员输入原密码、新密码和确认密码，修改成功后清除当前标签页登录态并返回管理员登录页。

## 范围

- 管理端桌面侧栏账号区域与移动端抽屉账号区域增加“修改密码”入口。
- 使用 Element Plus 对话框承载表单，不新增独立路由。
- 后端新增管理员专用密码修改接口，复用现有 bcrypt 校验与密码更新逻辑。
- 新密码长度为 12–64 个字符，UTF-8 编码不得超过 bcrypt 的 72 字节限制。
- 新密码必须不同于原密码；前端要求两次输入一致。
- 修改成功后仅退出当前标签页会话，其他设备已签发的 JWT 最长仍可使用至当前 24 小时有效期结束。

## 接口与数据流

前端调用 `POST /api/auth/admin/change-password`，请求体为：

```json
{
  "current_password": "现有密码",
  "new_password": "至少十二位的新密码"
}
```

后端要求 Bearer Token 对应用户角色为 `admin`，验证原密码、密码差异和新密码规则后更新 `users.password_hash`，返回 `204 No Content`。失败时分别返回原密码错误、密码重复或参数校验错误；不暴露密码哈希和内部异常。

## 页面交互

管理员账号区提供带文字或明确无障碍标签的“修改密码”按钮。对话框包含原密码、新密码和确认密码，均支持显示/隐藏。提交期间禁用重复操作；成功后显示提示、关闭对话框、清理 Pinia 与 `sessionStorage` 中的管理员身份，并跳转 `/admin/login`。关闭对话框时清空敏感输入。

## 验证

- 后端测试覆盖：管理员成功改密、错误原密码、游客越权、新密码少于 12 位、与原密码相同。
- 前端 TypeScript/Vite 构建通过。
- 浏览器验证桌面端和移动端入口可见、表单校验有效，成功后回到管理员登录页，旧密码失效、新密码可登录。

## 非目标

- 不增加忘记密码、短信/邮件找回或超级管理员代重置。
- 不新增跨设备 JWT 撤销或令牌版本字段。
- 不修改游客匿名身份流程。
